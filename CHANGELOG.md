### Unreleased (PianoTell fork)
- Harden `EditEmojiModal` save/delete promise chains. The previous code chained `.then(...)` with no `.catch(...)`, so any failure (e.g. the `DELETE /api/cache` call returning 409 because of a permission issue on `storage/cache`) surfaced as "Unhandled Promise Rejection: [object Object]" in the browser console and left the submit button stuck in a loading state. Now: errors during save/delete go through the standard `Modal.onerror` flow; a failed cache-clear is treated as a non-fatal warning (the formatter cache is keyed and rebuilds itself on next render); and `this.loading` is always reset in a final `then`.
- Prevent the upstream `the-turk/flarum-flamoji` extension from coexisting with this fork. Declares a composer-level `conflict` on the upstream package so `composer require` fails fast, and ships a one-shot migration that (a) strips `the-turk-flamoji` out of the `extensions_enabled` setting if present (safety net for sites that bypass composer), and (b) deletes the ~12 orphan `the-turk-flamoji.*` rows Flarum keeps in the `settings` table after an extension is uninstalled. Both extensions share the `custom_emojis` table, so emoji data carries over without action.
- Replace the `ListEmojisController` "limit equals default → return all rows" hack with proper JSON:API pagination plus an explicit `?filter[all]=1` escape hatch. The forum picker (which needs the full custom-emoji set in one shot) and the admin "Export to JSON" flow now pass `filter[all]=1`. All other callers get a normally paginated response.
- Switch `ImportEmojiController` to a plain `RequestHandlerInterface` returning `204 No Content` instead of extending `AbstractCreateController`. The previous handler returned only the *last* imported emoji as a JSON:API resource — the JS caller already ignores the body, and JSON:API's "create" semantics don't fit a bulk-import endpoint. `ImportEmojiHandler::handle()` now also returns `void`.
- Extract the inline ~80-line picker click handler in `forum/index.js` into named `buildPicker` / `onPickerButtonClick` helpers, and add an `extend(TextEditor.prototype, 'onremove', ...)` cleanup that destroys the EmojiButton picker and removes the injected `flamojiContainer` `<div>` when the editor is unmounted. Without this every composer open/close cycle leaked a DOM node and an EmojiButton instance with its event listeners.
- Code hygiene: fix the wrong `use PianoTell\Flamoji\Emoji` (real model is at `Models\Emoji`) in `EmojiSerializer`, combine the two `extend(TextEditor.prototype, 'toolbarItems', ...)` blocks in `forum/index.js`, and add the missing `import app from 'flarum/common/app'` to admin components that were relying on the global.
- Scope the flamoji button lookup to the editor element instead of the whole document. Previously `document.querySelector('.Button-flamoji')` always returned the first match in the DOM, so when more than one composer/editor was on the page (e.g. a discussion reply box plus a separate reply form) the picker would attach to the wrong editor's button. Now uses `this.element.querySelector(...)` inside `TextEditor.oncreate`.
- Use the modern `flarum/common/components/*` and `flarum/common/utils/*` import paths in `EditEmojiModal` (was using the deprecated `flarum/components/*` / `flarum/utils/*` aliases).
- Removed the dead `js/src/common/index.js` (its only contents were a stray `console.log` and an initializer with the wrong extension ID `pianotell/flarum-ext-flamoji`). Removed the matching `export * from './src/common'` re-exports from `js/admin.js` and `js/forum.js`.
- Migrated default settings from a one-shot `addSettings` migration into runtime `Extend\Settings()->default(...)` declarations in `extend.php`.Fresh installs no longer rely on the seed migration; existing installs are unaffected (rows already in `settings`). Removed `migrations/2021_07_06_000000_set_default_settings.php`.
- Bumped compatibility constraints to `flarum/core: ^1.8.0` and `php: >=8.0`to reflect what we actually target and test against. (Was `flarum/core: ^1.2.0` with no PHP constraint.)
- **Security:** require admin permission on create/update/delete/import emoji API routes.Previously any authenticated user could mutate the custom-emoji table. The list route remains public (the forum picker uses it client-side).
- **Security:** HTML-escape emoji `title` and `path` in `ConfigureTextFormatter` to prevent admin-stored XSS through every rendered post.
- **Validation:** reject empty `textToReplace`/`path` and trigger text containing whitespace, throwing a Flarum `ValidationException` (HTTP 422). Prevents one bad row from breaking the formatter cache rebuild.
- Fix PHP 8.2+ "Creation of dynamic property" deprecation in `ConfigureTextFormatter` (declared `protected UrlGenerator $url`). The deprecation, when surfaced to output, was triggering a fatal `Laminas\HttpHandlerRunner\Exception\EmitterException` ("Output has been emitted previously; cannot emit response") on every page that rendered posts.
- Fix `DeleteEmoji` command never actually deleting anything: constructor was assigning to a typo'd dynamic `$tagId` property instead of the declared `$emojiId`, so `DeleteEmojiHandler` always called `Emoji::findOrFail(null)`.
- Renamed package to `pianotell/flarum-ext-flamoji` (fork of the discontinued [`the-turk/flarum-flamoji`](https://github.com/the-turk/flarum-flamoji)).
- Renamed PHP namespace `TheTurk\Flamoji` → `PianoTell\Flamoji`.
- Renamed extension ID `the-turk-flamoji` → `pianotell-flamoji` (settings keys, locale root, asset path).
- Renamed API route prefix `/the-turk/...` → `/pianotell/...`.
